# ТЗ — Мобильное приложение учёта рабочего времени по распознаванию лица

**Рабочее название:** FaceClock
**Дата:** 2026-07-10
**Статус:** черновик v0.1 (на согласование)

---

## 1. Цель и краткое описание

Мобильное приложение, которое фиксирует **приход** и **уход** сотрудника с работы
через камеру телефона с распознаванием лица. Отметка возможна только когда
сотрудник действительно физически находится на рабочем месте, а перед камерой —
живой человек (а не фото/видео/маска).

Ключевая ценность: убрать бумажные табели и «отметки за друга», дать бизнесу
достоверный учёт времени и прозрачную отчётность.

---

## 2. Роли пользователей

| Роль | Возможности |
|------|-------------|
| **Сотрудник** | Отметка прихода/ухода, просмотр своей истории и отработанных часов |
| **Руководитель / HR** | Табель по подразделению, аномалии, ручное подтверждение/отклонение спорных отметок |
| **Администратор** | Управление сотрудниками, рабочими местами (геозоны), политиками, устройствами |

---

## 3. Основные сценарии (user stories)

1. **Регистрация лица (энролмент).** Сотрудник один раз проходит съёмку лица
   (несколько ракурсов) — создаётся биометрический шаблон. Делается очно, под
   контролем HR, чтобы гарантировать, что шаблон принадлежит нужному человеку.
2. **Приход.** Сотрудник открывает приложение → «Отметить приход» → наводит лицо
   в рамку → проходит проверку живости → сервер подтверждает совпадение и
   геолокацию → фиксирует время прихода.
3. **Уход.** Аналогично, кнопка «Отметить уход». Считаются отработанные часы.
4. **История.** Сотрудник видит свои смены, часы, статусы отметок.
5. **Табель и аномалии.** Руководитель видит сводку, флаги подозрительных отметок,
   может подтвердить/отклонить.

---

## 4. Функциональные требования

### 4.1 Сотрудник
- Вход по логину/паролю или SSO компании + привязка устройства.
- Экран «Отметиться» с крупными кнопками «Приход» / «Уход» и статусом текущей смены.
- Съёмка лица с индикацией: рамка-овал, подсказки («моргните», «поверните голову»).
- Мгновенный результат отметки (успех/ошибка + причина).
- История отметок с фильтром по датам, суммарные часы за период.
- Push-уведомления (напоминание отметиться, если смена началась/закончилась).

### 4.2 Руководитель / HR
- Табель: список сотрудников, статус (на работе / ушёл / не пришёл), часы.
- Лента отметок с фото-подтверждением и метками риска.
- Очередь спорных отметок → подтвердить / отклонить с комментарием.
- Экспорт табеля (CSV/XLSX) за период.

### 4.3 Администратор
- CRUD сотрудников, приглашения, повторный энролмент лица.
- Рабочие места: адрес, координаты, радиус геозоны, Wi-Fi SSID / BLE-маяки.
- Политики безопасности (см. раздел 6): что обязательно, что опционально.
- Управление устройствами (сброс привязки при смене телефона).

---

## 5. Нефункциональные требования
- **Платформы:** Android и iOS (единая кодовая база — см. раздел 8).
- **Скорость отметки:** ≤ 3 сек от наведения до результата при нормальной сети.
- **Офлайн-режим:** отметка ставится в очередь и досылается при появлении сети
  (с сохранением серверного времени момента съёмки через доверенный источник).
- **Приватность/GDPR/152-ФЗ:** явное согласие на обработку биометрии, шифрование
  шаблонов, право на удаление, хранение только шаблонов (не сырых фото по умолчанию).
- **Безопасность:** TLS, шаблоны лица — только на сервере, никаких «сырых» биометрий
  на клиенте дольше сессии съёмки.
- **Доступность:** крупные элементы, голосовые/текстовые подсказки, контраст.

## 6. Защита от недобросовестных отметок ⭐ (ключевой раздел)

Проблема: сотрудник хочет отметить приход/уход, **не находясь на работе**
(отметиться из дома, «за друга», подсунуть фото/видео). Меры защиты —
многоуровневые, комбинируются по политике компании.

### Уровень 1 — «Это живой человек, а не подделка»
- **Liveness detection (проверка живости):**
  - *Пассивная* — анализ текстуры кожи, бликов, глубины (защита от фото на экране/бумаге).
  - *Активная* — случайное задание: «моргните», «поверните голову», «улыбнитесь».
    Задание выдаёт **сервер случайным образом** каждый раз → нельзя переслать
    заранее записанный ролик.
- **Anti-replay / одноразовый nonce:** каждая съёмка привязана к одноразовому
  токену от сервера с коротким сроком жизни → повторно отправить старый кадр нельзя.
- **Сравнение лица на сервере**, а не на клиенте — клиенту нельзя доверять.

### Уровень 2 — «Это именно ты, и с твоего устройства»
- **Привязка устройства:** один сотрудник — одно доверенное устройство.
  Отметка с чужого телефона блокируется / требует повторной авторизации.
- **Детекция дубля лица:** один и тот же человек не может отметить двоих
  (проверка, что лицо не совпадает с шаблоном другого сотрудника в этой же сессии).

### Уровень 3 — «Ты физически на рабочем месте»
- **Геозона (GPS):** отметка разрешена только в радиусе рабочего адреса.
  Флаг при попытке mock-локации / root/jailbreak.
- **Привязка к сети/маякам:** телефон должен видеть корпоративный **Wi-Fi SSID**
  или **BLE-маяк** на территории (сложно подделать удалённо).
- **Динамический QR/NFC на проходной:** на рабочем месте висит экран/метка с
  QR-кодом, который меняется каждые N секунд; сотрудник сканирует его при отметке.
  Комбинация «лицо + свежий QR» практически исключает удалённую отметку.
- **Серверное время:** момент отметки берётся с сервера, а не с телефона
  (защита от подмены времени на устройстве).

### Уровень 4 — «Контроль и разбор»
- **Фото-аудит:** кадр отметки сохраняется (по политике/согласию) для ручной сверки.
- **Скоринг риска:** каждой отметке присваивается risk-score (вне геозоны, низкий
  liveness, новое устройство, ночное время и т.п.).
- **Очередь аномалий:** подозрительные отметки не засчитываются автоматически —
  уходят руководителю на подтверждение.
- **Журнал аудита:** неизменяемый лог всех отметок и решений.

> **Рекомендуемый минимум для старта:** активная liveness + серверное сравнение +
> геозона + серверное время + risk-score. QR/BLE и привязка устройства —
> подключаются по мере необходимости (для мест с высоким риском махинаций).

## 7. Модель данных (верхнеуровнево)
- **Employee**: id, ФИО, подразделение, статус, faceTemplateId, deviceId, согласие.
- **Workplace**: id, адрес, координаты, радиус, wifiSSID[], bleBeacons[].
- **Attendance (отметка)**: id, employeeId, type(in/out), serverTime, geo, livenessScore,
  riskScore, status(confirmed/pending/rejected), photoRef, workplaceId.
- **Device**: id, employeeId, platform, pushToken, trusted.
- **AuditLog**: кто, что, когда, результат.

---

## 8. Технологический стек (предложение к согласованию)
- **Мобильный клиент:** React Native (Expo) или Flutter — одна кодовая база на iOS/Android.
- **Камера/лицо на клиенте:** только детекция лица и наведение (ML Kit / Vision).
- **Распознавание и liveness:** серверный сервис (например, свой на основе
  открытых моделей, либо облачный SDK) — итоговое решение принимает сервер.
- **Бэкенд:** REST/GraphQL API (Node.js/NestJS или Python/FastAPI), PostgreSQL,
  хранилище фото (S3-совместимое) с шифрованием.
- **Уведомления:** FCM (Android) / APNs (iOS).

*Стек финализируем после выбора дизайна и подтверждения требований.*

## 9. Этапы (roadmap)
1. **MVP:** энролмент, приход/уход, liveness (активная), геозона, серверное время,
   история сотрудника, простой табель для HR.
2. **v1:** risk-score, очередь аномалий, экспорт табеля, push-напоминания.
3. **v2:** QR/BLE-привязка, привязка устройства, офлайн-режим, детекция дублей,
   расширенная аналитика.
